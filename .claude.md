## What We're Building
Elmi is a zero-friction Scope 3 data ingestion portal for boutique ESG consultants. Suppliers upload utility bills via anonymous Magic Links; AI extracts structured data; consultants verify and export to carbon calculators. We are NOT building a carbon calculator. We are middleware.

## Stack (Non-Negotiable)
| Layer | Technology |
|---|---|
| Framework | Next.js 16 (App Router) + React 19 |
| UI | Mantine v8 — NO Tailwind, NO shadcn, NO other component libraries |
| ORM + AuthZ | ZenStack v3 (Kysely engine) — schema in `zenstack/schema.zmodel` |
| Auth | Better-Auth (self-hosted, data in own Postgres) |
| AI Ingestion | Gemini 2.5 Flash via Vercel AI SDK *(model TBD — evaluate cost/quality)* |
| AI Validation | Claude Sonnet via Vercel AI SDK *(model TBD — GPT-4o mini and other cheap models worth testing)* |
| Background Jobs | Inngest |
| Database | PostgreSQL on Google Cloud SQL |
| File Storage | Google Cloud Storage (signed URLs for access) |
| Hosting | Vercel (compute) + GCP (data) |

## Architecture Rules

### Next.js Patterns
- Server Components and Server Actions are the default for ALL data fetching and mutations
- NEVER use the `pages/` directory — App Router only
- No tRPC, no GraphQL, no REST API layer — Server Actions handle mutations, Server Components handle reads
- Co-locate files: keep components, actions, and types near the feature they serve

### Mantine Rules
- Use Mantine components exclusively for UI: `Stack`, `Group`, `Box`, `Text`, `Button`, `Table`, `Modal`, `Dropzone`, etc.
- Use Mantine hooks: `useForm`, `useDisclosure`, `useDebouncedValue`, `useMediaQuery`
- NEVER write `className="flex"` or any Tailwind utility — use Mantine's `style` prop or CSS Modules
- Mark files `"use client"` only when using Mantine hooks or browser APIs

### ZenStack v3 Rules
- ALL models defined in `zenstack/schema.zmodel` — never create raw Prisma models
- Every model MUST have explicit `@@allow` and `@@deny` policies — no model ships without access control
- Run `npx zenstack generate` after every `zenstack/schema.zmodel` change before running migrations
- Do NOT install or import `@prisma/client` directly — use ZenStack's generated client
- Multi-tenant isolation: consultants NEVER see other consultants' data — enforce at the schema level, not in application code
- Magic Link sessions: scoped to a single project/supplier pair — enforce in schema

### Better-Auth Rules
- Auth config lives in `lib/auth.ts`
- Use the ZenStack adapter (`@zenstackhq/better-auth`)
- Organizations plugin is enabled — all consultant accounts belong to an org
- Magic Links for supplier upload sessions — no account creation required for suppliers
- Never store auth secrets in code — environment variables only

### Inngest Rules
- All AI extraction jobs run through Inngest — never call Gemini/Claude directly from a Server Action if the operation could exceed 60 seconds
- Event naming convention: `elmi/document.uploaded`, `elmi/extraction.completed`, `elmi/validation.completed`
- Each Inngest function handles ONE pipeline step — no monolithic extraction functions
- Inngest functions live in `inngest/functions/`

### AI Extraction Pipeline
Two-stage, cross-provider validation:
1. **Stage 1 — Extraction:** PDF → images → structured JSON with per-field confidence scores
2. **Stage 2 — Validation:** Original image + Stage 1 JSON → validation report flagging discrepancies
3. **Stage 3:** Consultant split-screen review

Both model calls go through the Vercel AI SDK. Model identifiers are environment variables — never hardcoded.

> **Model selection is NOT final.** Evaluate cost vs. quality before committing:
> - Stage 1 candidates: Gemini 2.5 Flash, GPT-4o mini, Claude Haiku
> - Stage 2 candidates: Claude Sonnet, GPT-4o mini, Gemini Flash
> - The Vercel AI SDK makes swapping providers trivial — keep model IDs in env vars.

### Google Cloud Storage
- Raw PDFs stored in GCS — never in Postgres
- Use signed URLs (short TTL) for serving PDFs in the verification workspace — never proxy through the app server
- File path convention: `{orgId}/{projectId}/{supplierId}/{documentId}/{filename}`

### TypeScript & Validation
- Strict mode always — `"strict": true` in tsconfig
- No `any` — use `unknown` and narrow, or define proper types
- Prefer `type` over `interface` for data shapes; `interface` for things that extend
- Validate all Server Action inputs with Zod — never trust client data
- Use Zod schemas to type AI extraction outputs (structured output via Vercel AI SDK)

## Multi-Tenant Data Model
Two user types — keep them clearly separated:
- **Consultants:** Authenticated Better-Auth users belonging to an org. Own projects and see their org's data only.
- **Suppliers:** Anonymous Magic Link sessions. Can only upload to their specific assigned project. No dashboard access.

## Feature Areas
- **Consultant Command Center:** Dashboard (supplier status table), Verification Workspace (split-screen PDF + extracted data), Export Manager (CSV export to CYF/Aclymate/Persefoni)
- **Supplier Portal:** Single-page Magic Dropzone (no login), post-upload Instant Scorecard

## File Structure & Co-location
```
app/
  (consultant)/              # Consultant Command Center routes
    dashboard/
      page.tsx               # Server Component (data fetching)
      _components/           # Co-located client components
      actions.ts             # Co-located Server Actions
    verification/
    export/
  (supplier)/                # Supplier Portal routes
    [token]/                 # Magic Link dynamic segment
  api/
    auth/[...all]/           # Better-Auth catch-all handler
    inngest/                 # Inngest webhook handler
inngest/
  client.ts
  functions/                 # One file per pipeline step
lib/
  auth.ts                    # Better-Auth config
  db.ts                      # ZenStack enhanced client
  gcs.ts                     # GCS utilities
  validators/                # Shared Zod schemas
zenstack/
  schema.zmodel              # Single source of truth for data model
```

## Naming Conventions
- Server Action files: `actions.ts` co-located with the route
- Co-located components: `_components/` directory (underscore prefix = not a route)
- Inngest events: `elmi/<resource>.<action>` (e.g., `elmi/document.uploaded`)
- GCS paths: `{orgId}/{projectId}/{supplierId}/{documentId}/{filename}`
- Zod validators: `lib/validators/<domain>.ts` for shared schemas; inline for route-specific

## What NOT to Do
- Do NOT suggest Tailwind, shadcn, Radix, or any non-Mantine UI solution
- Do NOT suggest tRPC, GraphQL, or REST endpoints for internal data fetching
- Do NOT suggest Prisma standalone (ZenStack wraps it)
- Do NOT suggest Clerk, Auth.js, or Supabase Auth (Better-Auth is decided)
- Do NOT suggest AWS (we're on GCP)
- Do NOT create API routes for things that can be Server Actions
- Do NOT call AI models synchronously from Server Actions for multi-page documents
